# PaperInsight 2.0 需求基线（归档整理版）

> 说明：这份 2.0 文档按“当前仓库里仍能核查到的事实”重新整理，不再把今天无法验证的旧设想写成既成事实。

## 1. 2.0 仍可核查到的设计基线

从当前仓库反推，2.0 阶段确立的几个基础目标今天仍然清晰可见：

- 用命令行工具批量处理一个目录里的 PDF。
- 提供“联网增强”和“本地兜底”两条运行路径。
- 对每个 PDF 建立缓存，避免重复解析。
- 处理失败的文件不阻塞整批任务，并记录错误日志。
- 最终输出按影响因子排序的 Excel 报表，并可选重命名 PDF。

## 2. 在当前仓库中还能看到的 2.0 能力映射

### 2.1 CLI 引导式启动

当前 `paperinsight/cli.py` 仍保留了 2.0 风格的启动顺序：

1. `check`
2. `config`
3. `analyze`

这说明“先检查环境、再配置、最后分析”的流程基线仍然有效。

### 2.2 双模式运行

当前仓库仍然保留：

- `api` 模式：启用 LLM / 联网能力
- `regex` 模式：本地正则兜底
- `auto`：按配置自动选择

这与 2.0 想解决“既要高精度，也要有免费兜底”的目标一致。

### 2.3 MD5 缓存

当前 `CacheManager` 仍以 PDF 的 MD5 为索引，缓存结构化结果和 Markdown 文本；这正是早期“断点续跑、避免重复调用”的核心思路。

### 2.4 错误隔离

当前 `AnalysisPipeline` 在批处理时会把失败项单独写入错误列表和 `error_log.txt`，不会因为单篇论文失败而中止整批任务。

### 2.5 Excel 导出与重命名

当前实现仍保留：

- 报表导出
- 按影响因子排序
- PDF 重命名模板

这三项可以视为 2.0 目标在当前仓库中的持续落地。

## 3. 与当前实现相比，2.0 叙事已经过时的地方

为避免误导，这里明确列出已经变化的部分：

- 当前最低 Python 版本要求是 3.9，而不是 3.8。
- 当前主链路更强调 MinerU + 清洗器 + LLM + Pydantic，而不是早期的 PaddleX-first 叙事。
- 当前报表默认文件名是带时间戳的 `论文分析报告_*.xlsx`，不是固定文件名。
