# PaperInsight v3.0 设计与实现对照

> 说明：本文不是“愿景版 PRD”，而是按当前仓库实现重新整理的 v3.0 核心设计与落地映射。

## 1. v3.0 的核心升级方向

当前仓库能直接证明的 v3.0 升级重点有四个：

1. 用 MinerU 作为优先 PDF 解析器。
2. 在 LLM 前增加文本清洗步骤，缩短输入并保留高价值块。
3. 使用嵌套式 `PaperData` / `PaperInfo` / `DeviceData` / `OptimizationInfo` / `DataSourceReference` 数据模型。
4. 把最终导出统一收敛到 Excel/JSON 报表与错误日志。

## 2. 当前实现映射

### 2.1 PDF 解析层

- `paperinsight/parser/mineru.py` 提供 `cli` 与 `api` 两种 MinerU 调用方式。
- `paperinsight/core/pipeline.py::_parse_pdf()` 会优先尝试 MinerU，失败时回退到基础 PDF 文本提取。

### 2.2 文本清洗层

- `paperinsight/cleaner/section_filter.py` 已经不是简单章节截断，而是 Markdown block 切分、表格锚点化、块打分、邻域保留、预算内重组。
- `AnalysisPipeline._extract_from_parse_result()` 会先 `clean()`，再把清洗结果送入提取器。

### 2.3 结构化提取层

- `paperinsight/core/extractor.py` 会在 LLM 路径下构造统一 Prompt，并把结果校验为 `PaperData`。
- `paperinsight/models/schemas.py` 定义了嵌套模型和 Excel 扁平化逻辑。
- OpenAI 路径当前会优先尝试严格 `json_schema` 输出；其他提供商走兼容路径。

### 2.4 IF 补全与报告导出层

- `paperinsight/core/pipeline.py::_supplement_impact_factor()` 负责影响因子补全/校正。
- `paperinsight/core/reporter.py` 负责固定 20 列 Excel 导出与 JSON 导出。
- 错误日志由 `paperinsight/utils/logger.py` 和 `error_log.txt` 路径配合生成。

## 3. v3.0 在当前仓库中的可验证收益

- 单篇论文的多器件数据会保留在 `devices` 列表里，再按换行拼接写入 Excel。
- 原文溯源字段已固定为四列：`EQE Source`、`CIE Source`、`Lifetime Source`、`Structure Source`。
- 报表当前默认按影响因子降序排序。
- 同一篇 PDF 再次处理时，可直接命中 MD5 缓存。

## 4. 需要特别说明的边界

- 当前运行时默认 `mineru.mode` 是 `api`，不是 `cli`。
- 当前默认只导出 Excel；JSON 需要显式启用。
- 影响因子补全是 best-effort 机制，不应把所有结果都视为“官方最终值”。
