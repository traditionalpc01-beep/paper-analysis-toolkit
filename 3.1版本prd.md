# PaperInsight v3.1 设计与实现对照

> 说明：本文只保留当前仓库中已经能被代码或测试直接证明的 v3.1 变化点。

## 1. v3.1 当前已落地的三项主线

### 1.1 结构感知清洗器

当前 `paperinsight/cleaner/section_filter.py` 已实现：

- Markdown 块切分
- 尾部噪声章节裁剪
- 表格锚点化
- 块级评分
- 邻域窗口保留
- 预算内重组 `condensed_text`

这说明 v3.1 的清洗器已经从“章节白名单”演进成“结构感知 + 预算控制”的实现。

### 1.2 影响因子校正链路

当前 `paperinsight/core/pipeline.py::_supplement_impact_factor()` 已支持：

- 用 MJL 解析期刊元数据
- 用 MJL profile 获取 IF
- 在失败时回退到搜索抓取与 WOS Journals API
- 把状态写入 `impact_factor_status`
- 在 `correct_existing_impact_factor=true` 时按条件覆盖原值

### 1.3 双语输出

当前 CLI 与提取器已经形成完整闭环：

- `paperinsight/cli.py::analyze()` 会在 LLM 可用且用户未显式传参时询问是否启用双语输出
- `paperinsight/core/extractor.py::_ensure_bilingual_text_fields()` 会在提取后做一次双语后处理
- `paperinsight/core/reporter.py` 会把符合格式的双语文本写入 Excel，并支持富文本显示

## 2. 当前测试里能直接证明的 v3.1 行为

从现有测试可以直接核查到：

- `tests/test_v31_features.py` 验证了清洗器会保留高分块、表格锚点和邻域上下文，同时去掉 `References` 尾部噪声。
- `tests/test_v31_features.py` 验证了 IF 校正后会写入 `impact_factor_status`，并能标记 `NO_ACCESS`。
- `tests/test_v31_features.py` 验证了 CLI 会在每次运行前询问双语输出选项。
- `tests/test_reporting_outputs.py` 验证了报告文件名是 `论文分析报告_<时间戳>`，并验证了当前 Excel 头部字段集合。

## 3. 当前实现边界

- `impact_factor_status` 会保存在内部数据与 JSON 中，但当前默认 Excel 不导出该列。
- 双语输出只有在 LLM 路径可用时才真正生效；纯 `regex` 模式不会触发双语后处理。
- v3.1 的 IF 校正仍然是 best-effort，而不是“所有期刊都能命中官方数据”。
