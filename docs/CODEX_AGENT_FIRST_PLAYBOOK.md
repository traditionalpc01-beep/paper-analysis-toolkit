# Codex Agent-First 八步手册

这份文档把后续推荐的 Codex 交互方式固定下来，目标是让同一套方法可以重复复用，而不是每次重新组织提示词。

## Step 1：先让 Codex 画仓库地图

目标：
- 识别入口、主链路、关键边界、外部依赖点

建议提示词：

```text
请先阅读 README.md、AGENTS.md、paperinsight/cli.py、paperinsight/core/pipeline.py，
输出当前仓库的 agent-friendly 架构地图。
要求：
1. 标出入口、主流程、关键数据边界、外部依赖点
2. 标出哪些知识还只存在代码里、没有沉淀到 docs
3. 给出一个最小 harness 改造清单
不要改代码，先出方案。
```

预期产物：
- 架构地图
- 缺口清单
- 最小改造方向

## Step 2：把仓库补成 system of record

目标：
- 把知识从聊天记录转回仓库文档

建议提示词：

```text
基于刚才的分析，请把当前仓库补成 agent 可持续工作的知识底座。
优先新增或补全这些文档：
1. docs/ARCHITECTURE.md
2. docs/PIPELINE_STAGES.md
3. docs/AGENT_WORKFLOW.md
4. docs/QUALITY_GATES.md
5. docs/KNOWN_GAPS.md
要求：
- 只写和当前代码一致的内容
- 必须引用到真实文件路径
- 文档要适合后续 agent 渐进式读取
```

预期产物：
- 路由清晰的 docs
- 更短的后续提示词

## Step 3：先固化约束，不先堆功能

目标：
- 明确哪些行为必须稳定

建议提示词：

```text
请基于当前仓库，整理一份必须机械化验证的工程约束。
重点覆盖：
- pipeline 各阶段输入输出边界
- impact factor 补全规则
- reporter 导出列稳定性
- 配置默认值与兼容性
- 回归测试样例
输出为：
1. 约束清单
2. 对应应新增的测试/校验脚本/CI 检查项
先不实现，先设计。
```

预期产物：
- 质量门禁文档
- 测试与脚本 backlog

## Step 4：实现最小 harness

目标：
- 给仓库增加可重复执行的反馈回路

建议提示词：

```text
请实现第一批最小 harness 能力，按优先级从高到低：
1. 针对 pipeline 主流程的 smoke test / regression test
2. 一个可重复运行的诊断命令或脚本
3. 文档与代码一致性的基础检查
4. 导出列稳定性的测试
改完后请自测，并告诉我如何复用这些能力驱动后续开发。
```

预期产物：
- check script
- regression tests
- 文档一致性检查

## Step 5：以后每次都用任务模板发需求

目标：
- 降低模糊需求带来的返工

建议提示词模板：

```text
请在当前仓库实现一个小步功能，流程按下面执行：
1. 先阅读相关文件并复述现状
2. 写一个简短实现计划
3. 实现代码
4. 补测试
5. 自查风险和可改进点
6. 给出最终改动说明
相关范围：
- 文件：
- 目标：
- 非目标：
- 验收标准：
```

## Step 6：每轮结束都做 agent 自审

目标：
- 把“经验”变成“仓库规则”

建议提示词：

```text
请对本次改动做一次 agent-first 自审：
1. 哪些规则还只存在代码里，没有进入 docs
2. 哪些测试还缺
3. 哪些地方未来容易产生 AI slop
4. 哪些约束应该升级为自动检查
```

## Step 7：复杂需求先出 execution plan

目标：
- 把复杂问题切成可验收的小阶段

建议提示词：

```text
这是一个复杂改动，请先不要写代码。
请输出一个 execution plan，包含：
1. 目标与范围
2. 受影响模块
3. 分阶段实施步骤
4. 每阶段验收标准
5. 风险与回滚点
6. 哪些内容应先沉淀到 docs 再编码
```

执行阶段时再继续：

```text
现在执行第 1 阶段，只做最小改动并补齐测试。
```

## Step 8：最后做持续清理

目标：
- 防止 repo 逐渐重新退化为“只有人能读懂”

建议提示词：

```text
请为当前仓库补一套轻量的 agent garbage-collection 机制：
1. 扫描过时文档
2. 扫描重复工具函数
3. 扫描未被测试覆盖的关键规则
4. 扫描 reporter / schema / docs 描述不一致
输出：
- 可自动执行的检查脚本
- 对应文档说明
- 后续建议的 CI 挂载方式
```

## 推荐的固定节奏

1. 先 issue
2. 再 plan
3. 再实现
4. 再测试
5. 再文档化
6. 最后关闭 issue
